## 工程偏好

> source id: `profile.engineering`

- TypeScript 优先，前端默认函数式组件和 hooks。
- 先做出可用结果，再在关键路径打磨质量。
- 不为假想需求过度抽象，优先选择能尽快落地且便于维护的方案。
- 前后端都按功能边界拆分，保持 UI、业务、状态、API、服务、数据访问职责清晰。
- 核心业务逻辑和关键路径需要测试，测试框架跟随项目已有配置。
- C 端页面重视 SEO，后台管理页面不额外承担 SEO 成本。

## Agent 设计理解

> source id: `profile.agent`

他对 Agent 的核心判断是：Agent 不是更会聊天的机器人，而是围绕目标、上下文、工具调用、执行、验证和权限边界构建的工作系统。

一个可用 Agent 至少要能理解任务目标、读取上下文、选择工具、执行动作、检查结果，并在权限边界内和人协作。判断产品是不是 Agent，重点看它能不能交付可验证、可追踪的结果，而不是只输出建议。

## OpenClaw 与工作流

> source id: `profile.openclaw`

OpenClaw 是他自研的本地多 Agent 调度平台：主 Agent 根据任务类型分发给专业子 Agent，并把结果回流整合。它接入飞书工具、搜索、浏览器控制和本地开发工具链，用 Skill 和 Hooks 把 AI Agent 嵌入研发流程。

这个项目体现了他的关注点：多 Agent 编排、上下文压缩、工具权限、流程自动化、可验证交付和本地工程环境协作。

## Design Token Lab

> source id: `profile.design_lab`

Design Token Lab 是 `zhiqiang.chat` 中的站内工具页。它的产品判断是：审美判断应该先发生在真实 HTML 样张里，而不是先看变量表。用户先比较多套真实 UI 风格，满意后再提取为可复用的 `design-tokens.md`。

第一版服务当前博客技术栈：Next.js、React、Tailwind v4、CSS variables 和玻璃质感视觉系统。

## Skills 与 MCP 判断

> source id: `profile.skills_mcp`

他把 Skills 和 MCP 当成工程 Agent 的能力边界管理方式，而不是越多越好的插件列表。

Skills 更像“按场景加载的专业工作方式”，MCP 更像“让 Agent 接触外部系统的权限通道”。是否启用，取决于角色、触发场景、权限范围、风险和维护成本。
