## AI 应用开发理解

> source id: `profile.ai_understanding`

我理解的 AI 相关开发，不只是把模型 API 接到页面里，而是用 AI 和 vibe coding 改造开发工作流，快速交付业务需求，减少沟通成本、对接成本、联调成本和重复实现成本。

优秀模型会让开发事半功倍，但真正决定 AI 应用效果的，是开发者能不能围绕目标制定稳定的工作流：如何拆解任务、组织上下文、选择工具、定义交付物、验证结果、迭代反馈，并让每一个目标都有一套可执行的体系。

我看重 AI 带来的核心价值是提效：让开发者更快理解需求、更快完成实现、更快验证结果，也更快把想法变成可运行、可展示、可迭代的业务结果。

## Agent 与工作流判断

> source id: `profile.ai_agent_workflow`

我对 Agent 的定义比较务实：把几件事串成流，帮助用户完成一个目标，就可以称为 Agent。Agent 的定义很广，从串联两件事到串联一百件事都可以叫 Agent，但背后的工程复杂度完全不是一回事。

Agent 的重点不是形式，而是能不能帮助用户做成一个目标。它可以表现为聊天界面、后台任务、自动化工具、研发助手、工作流系统或产品里的某个能力模块。

我判断一个 Agent 是否有价值，会看它是否具备明确目标、上下文组织、工具调用、步骤编排、结果验证和边界控制，而不是只看它是否“像聊天机器人”。

vibe coding 让很多开发动作被大幅提速，所以 AI 方向的工程实践更应该关注如何快速形成可运行版本、快速验证业务价值、快速修正问题，而不是停留在概念讨论或过度设计。

## vibe coding 核心能力

> source id: `profile.vibe_coding_core`

我认为 vibe coding 最核心的能力，不是“会让 AI 写代码”，而是能把一个模糊目标变成清晰的执行流程、上下文材料、验收标准和迭代节奏。

真正拉开差距的是目标拆解能力、上下文组织能力、工程判断能力和结果验收能力。AI 可以快速生成内容，但它需要一个能判断方向、控制边界、发现问题、修正实现的人来把结果落成可用功能。

所以我的优势更接近“AI 时代的工程编排能力”：知道什么时候让 AI 扩展产出，什么时候收敛方案，什么时候停下来审查安全和上下文，什么时候快速跑通业务价值，什么时候再重构和沉淀。

## AI 辅助开发工作流

> source id: `profile.ai_development_workflow`

我使用 AI 开发时，会先说清楚目标，再写计划、审查计划、微调计划，然后把计划落地成可以执行的开发任务。

具体到开发过程，我会让 AI 先领取和理解任务，再制定开发计划，遵循项目规范，读取个人记忆和个人风格，结合当前仓库上下文产出实现。产出之后由人来验收，发现 bug 后继续微调，直到功能稳定。

对稳定功能，我倾向于把关键路径沉淀成测试工具或验证脚本，再提交代码。部署上，如果是自建 CI/CD，代码通过流水线后交给运维上线；如果是 Vercel、Netlify 这类第三方平台，则通常通过构建和部署流程自动上线。

这套流程的重点是：AI 负责提高产出速度，人负责目标判断、上下文约束、质量验收和最终责任。

## AI 开发风险控制

> source id: `profile.ai_risk_control`

我在 AI 生成代码时，最关注两类风险：安全问题和上下文理解错误。

安全问题包括密钥泄露、权限边界不清、后端接口暴露、数据库读写权限过大、用户输入未校验、第三方服务调用方式不当等。AI 能生成代码，但不能默认它理解当前项目的真实安全边界，所以关键接口、环境变量、权限和数据访问必须由人审查。

上下文理解错误是 AI 开发里更常见的问题。AI 可能误解已有架构、忽略项目规范、把临时方案当成正式方案，或者在没有足够上下文时补出看似合理但并不存在的事实。我的处理方式是让 AI 先读项目、先写计划、再执行，并通过构建、测试、日志和人工验收不断收敛结果。

我也意识到 AI 开发里的很多坑需要被持续记录。后续更适合把踩坑、根因、解决方式和适用边界沉淀成结构化知识，而不是只停留在临时对话里。这样下一次遇到相似问题时，可以直接检索历史经验，减少重复排查成本。

我理解的 AI 应用开发者，并不是传统开发者加一个模型调用接口，而是能力边界被放大后的产品工程角色。

传统开发者更容易把重心放在具体编码实现上；AI 应用开发者还需要关注目标拆解、上下文工程、工具编排、Agent 工作流、生成结果评估、风险控制和持续迭代。编码仍然重要，但它不再是唯一瓶颈。

AI 放大了个人能力，效率提升是肉眼可见的。一个开发者如果能同时理解业务目标、工程结构、AI 能力边界和交付流程，就能用更短路径完成从需求到上线的闭环。

我也有过两天内完成项目交付的经历。这个项目是通过 vibe coding 找到一个开源的 OpenRouter 中转类项目，再结合真实需求快速改造和落地完成的。这类快速交付能力适合说明：我能在 AI 辅助下快速完成需求理解、技术选型、开源方案筛选、前后端实现、部署验证和迭代修正，而不是只停留在局部代码生成。

## AIGC 工具开发经验

> source id: `profile.aigc_tooling`

我的 AI 实践更偏 AIGC 应用层和工具型产品，覆盖过生成 HTML、生成图片、生成视频等方向。

其中，生成 HTML 和生成图片是我认为更适合沉淀工具链的两类场景：它们不是简单调用模型返回结果，而是要围绕某个具体领域设计输入结构、生成流程、质量判断、结果预览、二次编辑和批量生产能力。

我做 AIGC 输出质量控制时，会先明确约定“不想要什么”，再决定主题色、视觉风格和输出边界。内容层面尽量使用用户提供的真实数据，而不是让模型完全自由发挥。真实数据越充分，生成结果通常越稳定、越贴近业务，也更容易达到可用标准。

对生成 HTML 来说，我没有一个死板的评价标准。好结果首先要好看、好用，要在样式、结构、布局、交互和动画上符合用户输入。如果用户输入本身质量不高，但系统仍然能输出一个可用且好看的结果，这时重点就不只是用户输入，而是底层 Prompt、约束和生成流程是否足够强。

对生成图片来说，我不强调自己掌握多少专业构图、比例、光影或品牌视觉术语。更重要的是，工具能不能把我想要的东西讲清楚、表达清楚，并把生成结果稳定引导到我能接受的方向。

这类经验说明我关注的不是“能不能生成”，而是如何让生成结果在真实业务场景里更稳定、更优质、更可控，并通过工具设计提升某个领域的产出效率。

这类场景说明 OpenClaw 不只是一个通用聊天助手，而是更接近围绕我个人工作方式搭建的研发协作系统：它理解我的偏好，能读取项目语境，也能把文档、测试、生成和研发流程串起来。

## Agent 设计理解

> source id: `profile.agent`

我对 Agent 的核心判断是：Agent 不是更会聊天的机器人，而是围绕目标、上下文、工具调用、执行、验证和权限边界构建的工作系统。

一个可用 Agent 至少要能理解任务目标、读取上下文、选择工具、执行动作、检查结果，并在权限边界内和人协作。判断产品是不是 Agent，重点看它能不能交付可验证、可追踪的结果，而不是只输出建议。

## Skills 与 MCP 判断

> source id: `profile.skills_mcp`

我把 Skills 和 MCP 当成工程 Agent 的能力边界管理方式，而不是越多越好的插件列表。

Skills 更像“按场景加载的专业工作方式”，MCP 更像“让 Agent 接触外部系统的权限通道”。是否启用，取决于角色、触发场景、权限范围、风险和维护成本。
